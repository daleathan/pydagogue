An algorithm for git push¶

We saw how git stores its object in The curious coder’s guide to git.

Now we know about how git stores its objects, we can work out how git knows which objects to copy when it does a push.

Paths through commits¶

Git commits form a graph. The commits are the nodes in the graph. Each commit (except the first) has one or more parents, stored as hash references in the commit object. The references to commit parents are the links between nodes that form the edges in the graph.

The output of git log --graph shows the sequence of commits as a graph, with commits as nodes and parent links as edges. Here’s some example output from the git log command with flags to make the graph structure more obvious:

[desktop]$ git log --graph --oneline --topo-order bce99da
*   bce99da Completed merge of trouble, fixing conflicts along the way
|\  
| * 015736b Changes in the trouble branch
* | 9b26718 Mainline work
|/  
*   b5cf59b Merge in the experiment
|\  
| * 2e06fce Trying something new
* | 16d0937 The mainline keeps moving
|/  
* 3dca4d2 I like this new name better
* 2241938 Luckily we submitted the paper before we realized
* a3239b9 Paper ready for submission to Science
* 407d747 Fruit of enormous thought
* f35239a First backup of my amazing idea

If we start at a particular commit, and then track back following only one parent for each commit, this is a path in the commit history.

See defining paths for a more formal definition of a commit path.

An algorithm¶

Something like this algorithm might do the job:

  1. Get the commit hash corresponding the branch we are going to push;
  2. Follow every commit path (see above)back from this commit, until we hit a commit hash (filename) that the remote has. All the previous commits on the path, that the remote does not have, are missing commits;
  3. For every missing commit get the corresponding tree (directory listing) object. If the tree object is not in the remote objects directory, add to the list of missing trees;
  4. For every missing tree read the tree directory listing. Find any blob (file) objects in the directory listing that are not in the remote objects directory, add to the list of missing blob objects [1];
  5. Copy all missing commit, missing tree and missing blob objects to the remote objects directory;
  6. Update the remote branch to point to the same commit as the local branch;
  7. Update the local record of the last known position of the remote branch to point to the same commit.

There’s a specific example of git push at git push – synchronizing repositories. Here is how that example would follow our algorithm:

  1. We look up the hash for master, and we get 3ed451c8b4df1e43d190c6fc6bd42bf3054cd360 (abbreviated as 3ed451c);
  2. We follow all commit history paths back from 3ed451c to check for missing commits. We start with 3ed451c. The remote does not have a matching file in objects, so this is a missing commit. We only have one path to follow, because 3ed451c has only one parent – bce99da – and the remote does have a corresponding object, so we can stop looking for missing commits;
  3. We only have one missing commit, 3ed451c. We look in the contents of 3ed451c to find the tree object hash. This is 7f0de8dd99ee5c9987309668b5835796dc9c98c5. We check for this object in the remote objects directory, and sure enough, it is missing. We add this tree to the list of missing trees;
  4. We only have one missing tree – 7f0de8dd99ee5c9987309668b5835796dc9c98c5. We look in the contents of this tree object and check in the remote object directory for each object in this listing. The only missing object is bfb83ae9365ab2a8343ef5f3cb77f834bdffff93;
  5. We copy the objects for the missing commits (bce99da1c301714aa7d3d895c1ef2aca63ada629), missing trees (7f0de8dd99ee5c9987309668b5835796dc9c98c5) and missing blobs (bfb83ae9365ab2a8343ef5f3cb77f834bdffff93) to the remote objects directory;
  6. We set remote refs/heads/master to contain the hash 3ed451c8b4df1e43d190c6fc6bd42bf3054cd360;
  7. Set the local refs/remotes/usb_backup/master to contain 3ed451c8b4df1e43d190c6fc6bd42bf3054cd360.

Footnotes

[1]You have probably worked out by now that git directory listings can have files (called “blobs”) and subdirectories (“trees”). When doing the copy, we actually have to recurse into any sub-directories to get needed file (“blob”) and subdirectory (“tree”) objects. But, you get the idea.
